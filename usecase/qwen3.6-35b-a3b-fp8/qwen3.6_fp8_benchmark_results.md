## 每卡显存四维分解（RTX 4090 24GB，TP4）

| 维度 | 每卡占用 | 说明 |
|---|---:|---|
| 模型权重 (FP8) | 8.65 GB | ×4 = 34.6GB ≈ 权重文件 34.89GB |
| Mamba cache | 5.17 GB | conv 0.12 + ssm 5.05；344 个请求槽，每请求固定 ~15.4MB |
| KV cache | 5.74 GB | K 2.87 + V 2.87，容纳 602416 tokens |
| CUDA graph | 0.83 GB | |
| 保留/通讯 | 2.04 GB | nvidia-smi 显示的余量 |

**每 token KV ≈ 10 KB**：`2(K,V) × 2(kv_heads) × 256(head_dim) × 5(注意力层) × 2B ≈ 10KB`，
`10KB × 602416 = 5.74GiB` 与日志吻合。`num_kv_heads=2 < tp=4`，KV 在 4 卡上复制。

## 结论

1. **总 token 吞吐（prefill+decode）满载恒定 ~5000-6000 tok/s**，与上下文长度无关（GPU 算力瓶颈）。输出吞吐随上下文变长而下降，因算力更多花在 prefill。
2. **并发增益随上下文急剧衰减**：8K 靠并发把输出吞吐拉到 4.4×，240K 几乎无增益（1.1×，单请求就已 prefill-bound 压满算力）。
3. **纯 decode 从 c1 到高并发掉得快**（8K:167→13 tok/s），多请求分摊 decode 算力；但 c1 各上下文纯 decode 都很快（111-167 tok/s），decode 本身不怕长上下文。
4. **高并发的 prefill存入时间主要是排队**：8K prefill 本身仅 0.29s，c67 的 38s 几乎全是排队。
5. **吞吐拐点 ≈ 最大真实并发的 ⅓~½**，超过后加并发只涨延迟不涨吞吐。

## radix cache 与稳定性（重要）

默认配置（radix cache 开启）在 KV 池被高并发打满时会触发 sglang 0.5.9 的分配器 bug：

```
ValueError: token_to_kv_pool_allocator memory leak detected!
full_available_size=1393, full_evictable_size=601023, size=602416
```

本机在 `8K/1K` 并发 `c60~c67` 打满池时触发，`SIGQUIT` 杀掉全部 TP worker。合成随机压测（零前缀命中）用 `--disable-radix-cache` 绕开、且略快；**生产环境有共享前缀时 radix 应保持开启**。详见 `qwen3.5-a3b-f8-start.md`。
